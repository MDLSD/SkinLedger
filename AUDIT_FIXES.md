# Итоги аудита — что исправить

Сводка по четырём аудитам (контроль доступа, входные данные, аутентификация/конфигурация, финансовые расчёты) от 2026-07-17. IDOR-уязвимостей и SQL-инъекций не найдено; ниже — всё, что требует правок, по убыванию приоритета.

## Высокий приоритет (ошибки с реальным эффектом)

### 1. Противоречивый дефолт `sellFxRate` — искажает прибыль валютных сделок
- **Где:** `src/lib/actions/deals.ts:73-74`
- **Суть:** `sellCurrency` по умолчанию наследует `buyCurrency`, а `sellFxRate` по умолчанию `1`. Для сделки в USD (buyFxRate = 90) без явного курса продажи: cost = 9000 ₽, revenue = 120 ₽ → профит −8880 ₽ вместо +1800 ₽.
- **Фикс:** дефолт `d.sellFxRate ?? d.buyFxRate` (раз валюта наследуется), либо требовать курс явно, когда валюта продажи не базовая.

### 2. Невалидированные даты периода → управляемый 500
- **Где:** `src/lib/deal-list.ts:84-90` → `src/app/app/deals/page.tsx:45-46`
- **Суть:** `from`/`to` из URL идут в `new Date()` без проверки; `Invalid Date` в `where.buyDate` роняет Prisma (`PrismaClientValidationError`). Пример: `GET /app/deals?period=custom&from=garbage`.
- **Фикс:** в `periodRange` принимать только `/^\d{4}-\d{2}-\d{2}$/` + `!Number.isNaN(d.getTime())`, иначе игнорировать параметр (паттерн «мусор → дефолт», как уже сделано для `sort`/`status`).

## Средний приоритет

### 3. Нет security-заголовков
- **Где:** `next.config.ts` (пустой)
- **Фикс:** секция `headers()`: `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY` (или CSP `frame-ancestors 'none'`); HSTS — на проде. Полный CSP — отдельно, с оглядкой на inline-скрипты Next.

### 4. IP для rate-limit из подделываемого `x-forwarded-for`
- **Где:** `src/auth.ts:51`, `src/lib/actions/auth.ts:20-21`
- **Суть:** при прямом доступе к приложению (без доверенного прокси) атакующий ротирует поддельные IP и обнуляет IP-ось лимита логина/регистрации.
- **Фикс:** на этапе деплоя — гарантировать trusted proxy, срезающий клиентские `x-forwarded-for`, либо брать IP из соединения. Связано с известным ограничением in-memory лимитера (`src/lib/rate-limit.ts`, уже задокументировано в коде — заменить на общее хранилище при деплое).

## Низкий приоритет (харднинг и косметика)

### 5. Нет верхних границ у числовых полей сделки
- **Где:** `src/lib/validation.ts:29-30, 37-39, 48-51`
- **Суть:** принимаются `buyPrice=1e300`, `quantity=2000000000` → мусор в Decimal-колонках, потеря точности в будущих агрегатах.
- **Фикс:** `.max(1e9)` для цен, `.max(10000)` для курса, `.max(100000)` для количества.

### 6. У пароля нет максимальной длины
- **Где:** `src/lib/validation.ts:11-19`
- **Фикс:** `.max(128)` в `registerSchema` и `loginSchema` (bcrypt всё равно усекает до 72 байт — это гигиена).

### 7. Float-ноль отображается как «+0 ₽»
- **Где:** `src/lib/deal-math.ts:54-56` (`formatMoney`, проверка `value > 0`)
- **Суть:** профит `4.4e-16` (математический ноль, артефакт float) показывается со знаком «+» и сортируется выше настоящего нуля.
- **Фикс:** порог `Math.abs(value) < 0.005 → 0` (или округление до центов) перед сравнением и форматированием.

### 8. Check-then-act без атомарности в update/delete сделки
- **Где:** `src/lib/actions/deals.ts:146-151, 176-181`
- **Суть:** проверка владения (`findFirst({id, userId})`) и запись (`update/delete` только по `id`) — два запроса. Практически не эксплуатируемо, но атомарный вариант строже и короче.
- **Фикс:** `deal.updateMany({ where: { id, userId }, data })` / `deleteMany` с проверкой `count`.

## Требования к этапу 4 (дашборд) — учесть при реализации, не сейчас

- **`profit()` не знает про статус** (`src/lib/deal-math.ts:4-12`): наивная сумма `profit(deal)` смешает торговую прибыль с потерями на выводе. Суммировать прибыль только по `status === "sold"`; потери `withdrawn_via_skin` — отдельной метрикой.
- **Единый код расчёта:** агрегаты считать через те же `profit()`/`buyCostBase()`, что и таблица, — не дублировать формулы в SQL, иначе цифры разойдутся.
- **База периода:** `periodRange` фильтрует по `buyDate`; для «прибыли за период» логичнее `sellDate` — выбрать осознанно и одинаково для списка и дашборда.
- Каждый будущий `aggregate`/`groupBy` — с `where: { userId }` из сессии.

## Не требует действий (зафиксировано для истории)

- **Зависимости:** 5 moderate-advisory, все в транзитивных dev/build-цепочках (`@hono/node-server` внутри prisma CLI, `postcss` внутри next) — в прод-рантайм не попадают. `npm audit fix --force` не запускать (даунгрейдит next до 9, prisma до 6). Обновляться в рамках текущих мажорных версий по мере выхода патчей.
- **Восстановление пароля:** флоу отсутствует. Когда появится — токен одноразовый, с TTL, инвалидация при смене пароля + инвалидация активных сессий.
- **Прод-секреты:** `AUTH_SECRET` для прода сгенерировать заново, dev-значение из `.env` не переносить.
