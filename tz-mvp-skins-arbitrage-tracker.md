# ТЗ: MVP веб-сервиса учёта арбитража скинов CS2

**Рабочее название:** SkinLedger (заменить на своё)
**Версия документа:** 1.0
**Цель MVP:** заменить эксель-таблицу арбитражника — учёт сделок купли/продажи скинов с автоматическим расчётом прибыли и наглядным дашбордом.

---

## 1. Общее описание

### 1.1 Проблема
Арбитражник покупает скины на сторонних площадках, продаёт в Steam или на других площадках, выводит средства через скины. Учёт ведётся вручную (Excel / заметки / нигде), из-за чего человек не знает точную чистую прибыль с учётом комиссий, курсов и «выводных» скинов.

### 1.2 Решение (MVP)
Веб-приложение, в котором пользователь вручную заносит сделки, а сервис автоматически считает прибыль по каждой сделке и агрегирует статистику за период.

### 1.3 Что НЕ входит в MVP (осознанно)
- Автоимпорт из Steam или сторонних площадок (никаких API-ключей и логинов сторонних сервисов)
- Мобильное приложение (только адаптивный веб)
- Сравнение цен между площадками в реальном времени
- Алерты и уведомления
- Платные тарифы и приём платежей
- Мультивалютный учёт с автоматическими курсами (в MVP — одна базовая валюта, курс вводится вручную при необходимости)

Это второй этап. MVP должен быть готов и отдан пользователям за 2–3 недели.

## 2. Технологический стек (рекомендация)

| Компонент | Выбор | Обоснование |
|---|---|---|
| Фронтенд + бэкенд | Next.js (App Router) + TypeScript | Один проект, нейросети отлично его генерируют, бесплатный деплой |
| БД | PostgreSQL (Neon / Supabase, бесплатный тариф) или SQLite на старте | Простая реляционная модель |
| ORM | Prisma | Читаемые схемы, удобно для вайбкодинга |
| Стили | Tailwind CSS + shadcn/ui | Быстрый приличный UI без дизайнера |
| Графики | Recharts | Простая библиотека для дашборда |
| Авторизация | Email + пароль (NextAuth / Supabase Auth). Вход через Steam (OpenID) — по желанию, можно во второй этап | Не блокировать MVP интеграциями |
| Хостинг | Vercel (бесплатный тариф) | Ноль затрат на старте |

Допустимая альтернатива: любой стек, который уверенно генерирует ваша нейросеть (например, React + FastAPI). Главное — не менять стек посреди разработки.

## 3. Роли и авторизация

- **Гость** — видит лендинг и страницы входа/регистрации.
- **Пользователь** — видит только свои данные. Никакого шаринга и команд в MVP.
- Регистрация: email + пароль. Подтверждение email в MVP не обязательно (можно добавить позже).
- Восстановление пароля — обязательно (через email).

## 4. Модель данных

### 4.1 User
| Поле | Тип | Примечание |
|---|---|---|
| id | uuid | PK |
| email | string | unique |
| password_hash | string | |
| base_currency | enum (RUB, USD, EUR) | валюта отчётов, по умолчанию RUB |
| created_at | timestamp | |

### 4.2 Platform (справочник площадок)
Глобальный предзаполненный справочник + возможность добавить свою площадку.

| Поле | Тип | Примечание |
|---|---|---|
| id | uuid | PK |
| name | string | «Steam Market», «CS.Money», «Market.CSGO», «Buff163», «Skinport» и т.д. |
| default_sell_fee_pct | decimal | комиссия продажи по умолчанию, % (например, Steam ≈ 13 %) |
| default_buy_fee_pct | decimal | комиссия покупки по умолчанию, % (обычно 0) |
| is_custom | boolean | true — создана пользователем |
| user_id | uuid, nullable | владелец кастомной площадки |

Предзаполнить 6–8 популярных площадок с типовыми комиссиями. Комиссию всегда можно переопределить в конкретной сделке.

### 4.3 Deal (сделка)
Центральная сущность. Одна запись = один скин, проведённый по цепочке «купил → продал».

| Поле | Тип | Примечание |
|---|---|---|
| id | uuid | PK |
| user_id | uuid | FK |
| item_name | string | название скина, свободный ввод с автоподсказкой по прошлым сделкам пользователя |
| item_quality | string, nullable | Factory New / Minimal Wear и т.д. (выпадающий список + свободный ввод) |
| quantity | int | по умолчанию 1 (для однотипных дешёвых предметов) |
| buy_platform_id | uuid | FK → Platform |
| buy_price | decimal | цена покупки за единицу, в валюте покупки |
| buy_currency | enum | RUB/USD/EUR/CNY |
| buy_fx_rate | decimal | курс к базовой валюте, по умолчанию 1; вводится вручную |
| buy_fee_pct | decimal | подставляется из площадки, редактируется |
| buy_date | date | |
| status | enum | `holding` (куплен, ещё не продан) / `sold` / `withdrawn_via_skin` (использован как выводной) |
| sell_platform_id | uuid, nullable | FK → Platform |
| sell_price | decimal, nullable | цена продажи за единицу |
| sell_currency | enum, nullable | |
| sell_fx_rate | decimal, nullable | |
| sell_fee_pct | decimal, nullable | |
| sell_date | date, nullable | |
| withdrawal_discount_pct | decimal, nullable | для выводных скинов: потеря на выводе, % (см. 5.3) |
| note | text, nullable | свободный комментарий |
| created_at / updated_at | timestamp | |

### 4.4 Расчётные поля (не хранить, вычислять)
- `buy_cost_base` = buy_price × quantity × (1 + buy_fee_pct/100) × buy_fx_rate
- `sell_revenue_base` = sell_price × quantity × (1 − sell_fee_pct/100) × sell_fx_rate
- `profit` = sell_revenue_base − buy_cost_base (для status = sold)
- `margin_pct` = profit / buy_cost_base × 100
- `holding_days` = sell_date − buy_date

## 5. Функциональные требования

### 5.1 Управление сделками
- **Добавление сделки** — модальное окно или отдельная страница. Два сценария:
  1. «Купил» — заполняется только блок покупки, статус `holding`.
  2. «Купил и продал» — сразу оба блока, статус `sold`.
- **Закрытие сделки** — у сделки в статусе `holding` кнопка «Продано»: открывается форма блока продажи.
- **Редактирование и удаление** — любые поля, удаление с подтверждением.
- Форма обязана считать и показывать прибыль в реальном времени по мере заполнения полей («Прибыль: +342 ₽, маржа 12,4 %») — это ключевая «вау»-деталь.
- Автоподстановка комиссии при выборе площадки, с возможностью изменить в этой сделке.
- Автоподсказки названия скина из прошлых сделок пользователя (по подстроке).

### 5.2 Список сделок
- Таблица: скин, качество, площадка покупки → площадка продажи, цена покупки, цена продажи, прибыль (цветом: зелёный/красный), маржа %, дней в холде, статус, дата.
- Фильтры: период (пресеты: неделя, месяц, квартал, всё время + произвольный), статус, площадка, поиск по названию.
- Сортировка по любой колонке.
- Пагинация или ленивая подгрузка (50 строк на страницу).
- Экспорт текущей выборки в CSV.
- Импорт из CSV (шаблон с колонками из 4.3, страница с инструкцией и файлом-примером) — чтобы пользователь мог перенести свою эксель-таблицу за 5 минут. Это критично для конверсии первых пользователей.

### 5.3 Учёт выводных скинов
Специфика арбитража: часть скинов покупается не ради прибыли, а чтобы вывести баланс из Steam. В MVP решаем просто:
- Статус `withdrawn_via_skin`: пользователь указывает цену покупки в Steam (балансом) и фактически полученную сумму на внешней площадке.
- Разница считается как «стоимость вывода» и показывается отдельной строкой в статистике («Потери на выводе за период: −1 240 ₽»), не смешиваясь с торговой прибылью.

### 5.4 Дашборд (главная страница)
Карточки за выбранный период:
- Чистая прибыль (торговая)
- Оборот (сумма продаж)
- Средняя маржа, %
- Количество закрытых сделок
- Потери на выводе (отдельно)
- Сумма, замороженная в холде (стоимость покупки всех `holding`)

Графики:
- Прибыль по неделям/месяцам (столбчатый)
- Кумулятивная прибыль (линия)

Списки:
- Топ-5 самых прибыльных сделок за период
- Топ-5 убыточных
- Разбивка прибыли по площадкам продажи (таблица или donut)

### 5.5 Настройки
- Базовая валюта отчётов
- Управление своими кастомными площадками и их комиссиями
- Смена пароля, удаление аккаунта (с удалением всех данных)

### 5.6 Лендинг
Одна страница: заголовок с ценностью («Узнай, сколько ты реально зарабатываешь на арбитраже скинов»), 2–3 скриншота, список возможностей, кнопка регистрации. Плюс блок «Планируется» (автоимпорт, алерты) с полем для email — это дешёвая проверка спроса на второй этап.

## 6. Нефункциональные требования

- **Адаптивность:** полноценная работа с телефона (таблица сделок на мобиле сворачивается в карточки).
- **Язык интерфейса:** русский. Архитектурно не хардкодить строки намертво (i18n позже).
- **Скорость:** дашборд и список сделок < 1 с при 5 000 сделок у пользователя.
- **Безопасность:** пароли — только hash (bcrypt/argon2); все запросы к данным фильтруются по user_id на сервере; HTTPS; rate limit на логин.
- **Резервные копии БД:** автоматические (у Neon/Supabase есть из коробки — включить).
- **Валидация:** цены > 0, дата продажи ≥ даты покупки, комиссии 0–100 %.

## 8. План разработки (вайбкодинг, ~2–3 недели по вечерам)

| Этап | Содержание | Критерий готовности |
|---|---|---|
| 1 | Каркас проекта, авторизация, схема БД | Можно зарегистрироваться и войти |
| 2 | CRUD сделок + расчёт прибыли в форме | Можно завести сделку и увидеть прибыль |
| 3 | Список сделок: фильтры, сортировка, статусы | Удобно работать с 50+ сделками |
| 4 | Дашборд с картами и графиками | Видна картина за месяц |
| 5 | Импорт/экспорт CSV, выводные скины | Перенос из Excel за 5 минут |
| 6 | Лендинг, настройки, полировка мобильной версии | Не стыдно дать ссылку в чат |
| 7 | Деплой, свои реальные данные, 10–20 тестеров | Собран первый фидбек |

**Правило:** после этапа 2 начать вести в сервисе собственные реальные сделки. Всё, что раздражает лично вас, — приоритетный баг.

## 9. Метрики успеха MVP (первый месяц после запуска)

- ≥ 30 регистраций из тематических чатов/форумов (без платной рекламы)
- ≥ 10 пользователей, занёсших 10+ сделок (реальное использование, а не «зашёл посмотреть»)
- ≥ 5 содержательных фидбеков с запросами фич
- Главный сигнал для этапа 2: люди сами спрашивают про автоимпорт и готовность платить

7. Публичная страница статистики (шаринг результатов — вирусный канал)
